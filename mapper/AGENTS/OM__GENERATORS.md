## Architecture

### Two categories of generators

**Run generators** (Stage 6-8) produce edge package artifacts from a pipeline
run's mapping matrix. They read from `{run_dir}/` and write to
`{run_dir}/edge-package/`.

**Catalog generators** (one-time) produce reference data for target ontologies.
They read from external sources (NIEM API, OWL files, CMF XML) and write to
`specs/` inside the installed package. They run once per ontology version.

### Run generator pattern

Every run generator follows this structure:

1. `load_context()` to get `PipelineContext`
2. Load mapping matrix and concept inventory from run_dir
3. Load reference catalog from specs/ (if needed)
4. Generate artifacts deterministically from data
5. Write to `{run_dir}/edge-package/{subdirectory}/`

All behavior is data-driven. No domain-specific hardcoding. The generators
use `PipelineContext` for all derived names (namespaces, file prefixes).

### Catalog generator pattern

All three catalog generators produce identical output schemas:

| Output | Format | Purpose |
|--------|--------|---------|
| `{name}_reference_catalog_{version}.json` | JSON | Full type/property inventory with structural metadata |
| `{name}_catalog_summary_{version}.json` | JSON | Namespace-grouped summary |
| `{name}_type_directory_{version}.txt` | Text | One-line-per-type for scanning |

The catalog schema is defined in the Ontology Adapters domain
(`OM__ONTOLOGY_ADAPTERS.md`). All catalogs include: types with qname,
definition, baseType, pattern, properties, inheritanceChain,
propertyDefinitions; a propertyIndex; and an augmentationMap (if applicable).

### Edge package structure

Run generators collectively produce this directory tree:

```
edge-package/
  ontology/          # OWL/TTL modules (6a)
  cmf/               # CMF model (6a via owl_cmf_bridge)
  shapes/            # SHACL shapes (6a)
  vocab/             # SKOS codelists (6a)
  mappings/          # mapping-matrix.json, alignment-report.json, gap-analysis (6b)
  extensions/        # Extension definitions and catalog (6b)
  kg/                # Neo4j, RDF, SPARQL, import configs (6c)
  governance/        # Decision log, lineage, version manifest, validation report,
                     #   change-impact (6b, 8)
```

### Generation audit (pre-generation gate)

`generation_audit.py` runs at Stage 4 after `om-build-matrix`. Checks:

- GA-001: Active classes with zero source properties
- GA-002: Augment action on a non-NIEM flow
- GA-003: Augment entries with zero reuse-properties
- GA-004: Concepts with source properties but no property mappings
- GA-005: Missing structural scaffolding for extend/augment
- GA-006: Unresolved `human-must-decide` properties (blocks generation)
- GA-007: Scaffolding/targetType consistency — `augmentsType` must equal
  `targetType` for augment, `baseType` must equal `targetType` for extend
  (blocks generation)

Returns structured findings — the pipeline continues, but the operator
sees what needs attention.

## Contracts

### Generator inputs (from the orchestrator)

- `mapping-matrix.json` with `reviewStatus: "accepted"` on all mappings
- Property-level `action` values: `reuse-property` or `create-property`
- Structural scaffolding: `extensionType`/`baseType` for extend,
  `augmentationType`/`augmentsType` for augment

### Generator outputs (consumed downstream)

- Edge ontology OWL/TTL and CMF are both generated directly from the mapping matrix
- CMF is the canonical exchange artifact (generated by `MatrixToCmfBuilder` in `generate_cmf_from_matrix.py`)
- `package-manifest.json` has accurate stats matching actual artifact counts

### Catalog outputs (consumed by search and indexing)

- Reference catalogs must follow the exact JSON schema
- The `catalog_adapter.py` reads all three catalog formats identically
- `actions` and `typePatterns` keys define valid actions for a target ontology

## Development Rules

In addition to the hard rules in `AGENTS.md`:

- **Deterministic generation.** Generators produce identical output from
  identical input. No randomness, no LLM calls, no external API calls.
- **Data-driven, not domain-hardcoded.** Use `PipelineContext` and the
  mapping matrix — never hardcode ontology-specific logic in generators.
- **Only emit accepted mappings.** Generators read `reviewStatus: "accepted"`
  entries. Skip `pending-review` entries.
- **Catalog schema is the contract.** All three catalog generators must
  produce the same JSON structure. The catalog adapter treats them identically.

---

## Anti-Patterns

- **Hardcoded namespace URIs**: Use `PipelineContext.edge_namespace`,
  `PipelineContext.extension_namespace`, etc.
- **Skipping generation audit**: Always run `om-generation-audit` before
  Stage 6 to catch data integrity issues early.
- **Modifying input packages**: The edge package is a parallel artifact.
  Never modify the source ontology files.
